## Custom Project

### Dataset

The producer uses the original, unmodified `data/sales.csv` dataset included
with this project. It contains 178 sales transaction records. Each record
includes the following fields:

- `order_id` — unique UUID for the transaction
- `datetime` — ISO 8601 timestamp of the sale
- `region_id` — geographic region (e.g. `US-TX`, `CA-QC`)
- `currency_code` — transaction currency (e.g. `USD`, `CAD`)
- `product_id` — product SKU
- `unit_price` — price per unit
- `quantity` — number of units purchased
- `is_online` — whether the sale was made online
- `customer_id` — customer identifier
- `is_new_customer` — whether this is a first-time customer
- `device_type` — device used (desktop, mobile, tablet)
- `payment_method` — payment method used
- `referral_source` — how the customer found the product
- `discount_code` — discount code applied, if any
- `customer_note` — free-text note from the customer

The original dataset was used without modification.

### Kafka Messages

The producer reads each row from `data/sales.csv` and sends it as a separate
message to the Kafka topic `streaming-02-kafka-case`. The `region_id` field
is used as the Kafka message key, so all sales from the same region are
routed to the same partition and kept in order. No message fields were changed
from the original dataset.

### Consumer Processing

The consumer (`kafka_consumer_teja.py`) subscribes to
`streaming-02-kafka-case` and resets its offset to the beginning so it reads
all available messages on every run. It consumes up to 1000 messages or until
a 10-second timeout elapses with no new message — whichever comes first.

For each message received, the consumer:

- logs the full message dictionary to the console
- writes the complete row (all 15 original fields plus `_kafka_key`,
  `_kafka_partition`, and `_kafka_offset`) to
  `data/output/consumed_sales_teja.csv`
- increments a per-key counter keyed by `_kafka_key`

At the end of the run the per-key breakdown is logged in the summary.
No fields are filtered — the full raw message is written to CSV.

Added a per-Kafka-key message counter inside `consume_messages()`. Each
consumed message increments a `dict[str, int]` keyed by `_kafka_key`. The
final key-count breakdown is logged in `log_summary()`, making it easy to
see how many sales arrived from each region without inspecting the CSV
manually.

### Results

Running the producer sent 6 messages to the topic (3 per run across two
producer invocations). The consumer read 12 messages total because it
reset to the beginning and both sets of messages were still in the topic.
The per-key summary logged at the end showed:

```text
CA-ON = 2
CA-QC = 2
US-CA = 2
US-MO = 2
US-TX = 4
```

The output CSV `data/output/consumed_sales_teja.csv` contains all 12 rows
with all fields intact.

### Interpretation

- The producer and consumer are fully decoupled — the producer writes to the
  Kafka topic and exits; the consumer reads from it independently on its own
  schedule. They never call each other directly.
- Because the consumer resets its offset to the beginning, it can start late
  and still replay every message in the topic. This is what makes Kafka
  streams **replayable**, unlike a traditional queue that deletes messages
  after delivery.
- The message key (`region_id`) is not just a label — it determines which
  partition receives the message, keeping sales from the same region ordered
  together. This matters in production when you need per-region analytics or
  ordering guarantees.
- The per-key count immediately told a business story: US-TX had the highest
  sales volume (4 orders) while other regions had 2 each. A real pipeline
  could use this to trigger regional promotions or alert a sales team when
  a region goes quiet.
- Watching messages arrive one at a time — each with a partition number and
  an incrementing offset — made the mechanics of Kafka visible in a way that
  reading documentation alone cannot.

**What the producer sends.**

message to the Kafka topic. Every message uses the sale's `region_id`
(such as `US-TX` or `CA-QC`) as the Kafka key.

**What the consumer receives.**
The consumer subscribes to the topic, resets its offset to the beginning,
and reads each message one at a time. For every message it writes a row to
the output CSV and increments a counter for that message's Kafka key.

**What I learned from watching messages move through Kafka.**
The producer and consumer are fully decoupled - Kafka holds the messages in
between, so the consumer can start late and still replay the whole topic.
The message key is not just a label: it groups related records (here, sales
by region) and keeps them ordered within a partition. Counting by key made
the role of the key visible in the stream.
